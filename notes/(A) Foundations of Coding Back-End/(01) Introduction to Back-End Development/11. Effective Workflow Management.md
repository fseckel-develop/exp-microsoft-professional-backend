## Overview

- Workflow management organises, documents, and optimises development steps to keep projects running smoothly
- Comparable to a coordinated restaurant kitchen — efficiency prevents chaos
- Common methodologies: **Agile**, **Waterfall**, **Lean**, **Extreme Programming**
- Focus of this note: **Agile methodology**, workflow tools, version control, CI/CD.

---
## Agile Methodology

#### Purpose

- Emphasises iterative development, collaboration, and flexibility
- Uses short development cycles called **sprints** (typically 2 weeks)
- Produces a working product at the end of each sprint

#### Team Member Roles

- **Product Owner**:
    - Defines product vision, sets priorities, manages the backlog
    - Ensures delivered features provide user value

- **Scrum Master**:
    - Facilitates the agile process and ceremonies
    - Removes blockers and protects the team from disruptions

- **Development Team (usually 8 members)**:
    - Cross-functional team responsible for delivering sprint features
    - Collaborates to turn backlog items into working product increments

#### Agile Ceremonies

- **Sprint Planning**:
    - Held before each sprint
    - Team commits to specific high-priority backlog items

- **Daily Stand-Up**:
    - Short daily sync
    - Team reports yesterday’s work, today’s plan, and blockers

- **Sprint Review**:
    - Demonstrates completed work to stakeholders
    - Collects feedback and showcases product increments

- **Sprint Retrospective**:
    - Team reflects on the sprint
    - Identifies what went well, what needs improvement, and action items for the next sprint

#### Agile Artefacts

- **User Stories**:
    - Describe features from the end user’s perspective
    - Example: _“As an editor, I want to format text so that my articles are visually appealing.”_

- **Product Backlog**:
    - Prioritised list of desired features, enhancements, and fixes

- **Definition of Done (DoD)**:
    - Checklist that determines when work is considered complete
    - Usually includes passing automated tests, code reviews, and deployment to staging

## Version Control

- Version control tracks and manages changes to code or documents over time
- Essential for large technical projects with many contributors working in parallel
- Ensures simultaneous collaboration while maintaining a stable, functional product

#### Version Control Systems (VCS)

- Tools like **Git** store code, media, and project files in a centralised location
- Enable teams to:
    - Track revisions and daily work
    - Revert to previous versions
    - Share code without overwriting others
- Critical for distributed backend teams working on different system components

#### Branching

- Creating a separate “copy” of the code to modify without affecting the main codebase
- With that Developers can:
    - Experiment safely
    - Fix bugs
    - Test new features
- Once stable, branches are merged back into the main codebase
- If a merge introduces problems, teams can roll back to a stable version

## CI/CD Pipelines

- Automate integration, testing, and deployment of code
- Keep the codebase always deployable
- Catch issues early and shorten development cycles

- **Continuous Integration (CI)**:
	- Integrates code from multiple developers into a shared repository
	- Runs automated tests to confirm code correctness
	- Builds the program into a deployable state
	- Goal: detect issues early and keep the codebase always functional

- **Continuous Deployment/Delivery (CD)**:
	- Automates the steps required to release code to production
	- Includes:
	    - Preparing servers and databases
	    - Deploying code to live environments
	    - Monitoring systems after release
	- Ensures new features and fixes reach users quickly and reliably.

- Common Tools: **Jenkins**, **CircleCI**, **GitLab CI**

- Benefits:
	- Faster releases
	- Reduced manual work
	- Higher code reliability
	- Consistent deployment process
