## Overview

- GitHub supports collaboration across all stages of back-end development
- Enables teams to work simultaneously without overwriting each other’s work
- Provides tools for integration, review, and maintaining high-quality code

---
## Why GitHub Matters in Team Projects

- Prevents conflicts caused by isolated development
- Allows continuous visibility into teammates’ changes
- Ensures consistency through shared tools, libraries, and workflows
- Supports structured processes for coding, reviewing, testing, and building

---
## Common Actions in GitHub

#### Creating a Repository

- The repository is the “home base” for code and collaboration
- Steps:
    - Log into GitHub → Repositories → **New**
    - Add repository name + optional description
    - Choose **public** or **private**
- Example:
    - Creating a **private** repo named “Telemedicine App” to protect sensitive data.

#### Managing Branches

- Branches allow multiple developers to work safely in parallel
- Prevents unstable code from entering the main production-ready branch
- Naming convention usually includes the feature or fix (e.g., feature/virtual-consultations)
- Example:
    - Creating a branch dedicated to building the **virtual consultations** feature.

#### Submitting Pull Requests (PRs)

- Pull Requests merge changes from a feature branch into the main project with oversight
- Steps:
    - Commit changes → Go to **Pull Requests** tab → **New Pull Request**
    - Compare branches and provide a description of changes
- Example:
    - Submitting a PR to merge your virtual consultations feature into the main branch

#### Conducting Code Reviews

- Ensures code quality, functionality, and adherence to standards
- Reviewers can:
    - Leave comments on specific lines
    - Suggest improvements
    - Approve or request changes before merging
- Example:
    - Teammate reviews your PR, while you review theirs and flag missing data encryption in patient record management
