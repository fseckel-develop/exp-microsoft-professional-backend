## Understanding Core Project Files and Folders

- **Program.cs**
    - Entry point of most .NET applications
    - Execution begins here; other files run indirectly through it
    - Different project types may use different entry points

- **.csproj file**
    - Defines project configuration, dependencies, build rules
    - Written in XML with nested tags and attributes
    - Lists NuGet packages, target framework, and build metadata
    - Grows more complex as dependencies and linked projects increase

- **bin folder**
    - Stores compiled output (EXE, DLLs)
    - Contains binary files executed by the system
    - DLLs represent additional required code (e.g., libraries, packages)

- **obj folder**
    - Contains intermediate build files
    - Used during compilation but not part of the final output
    - Helpful for diagnosing build problems

## Best Practices for Organising Project Files

#### Structuring Code

- **Modularisation**
    - Split code into logical modules or categories
    - Organise by features (e.g., Users, Tasks, Comments)
    - Or organise by layers (e.g., Data, Business Logic, UI)

- **Separation of Concerns**
    - Each part of the code should handle a distinct responsibility
    - Feature-based: Keep all code for one feature together
    - Layer-based: Group files by purpose across the entire app

#### Naming Conventions

- **PascalCase**
    - Used for classes, methods, and public identifiers

- **camelCase**
    - Used for method parameters, private fields, local variables

- **Clear, descriptive names**
    - Prefer clarity over short names
    - Example: userAccountAge is better than num

## Refactoring and Documentation

- **Refactoring**
    - Improve structure and readability without changing behavior
    - Follow “Make it work → Make it right → Make it fast”
    - Clean code evolves over time; perfection is not required upfront

- **Documentation**
    - Include a **README** for project overview and onboarding
    - Use code comments when necessary for context or complex decisions
    - Avoid excessive comments that risk becoming outdated
    - Aim for self-explanatory code with comments for special cases
