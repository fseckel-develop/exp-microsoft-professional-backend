## What Are Classes?

- A class is a **blueprint/template** defining shared characteristics and behaviours
- Comparable to biological classification: class = group definition, object = individual
- Defines **properties** (data) and **methods** (behaviours/functions)
- Cannot perform actions until instantiated
- Supports **inheritance**, allowing subclasses to reuse and extend functionality

---
## Key Features of Classes

- **Properties**
    - Store data related to an object
    - Example: username, password, title, author, ISBN

- **Methods**
    - Actions the class can perform
    - Example: validate(), login(), borrow(), return()

- **Inheritance**
    - Child classes inherit and extend parent class features
    - Enables code reuse and organisation

---
## What Are Objects?

- An object is an **instance** of a class
- Represents a specific “thing” with a unique identity (e.g., memory address)
- Has **state** (current property values) and **behaviours** (methods it can execute)
- Multiple objects can be created from the same class, each with unique data
- Can only perform behaviours defined in its class (or inherited classes)
- Example:
	- **Book class** → properties: title, author, ISBN; methods: borrow(), return()
	- **Book object** → an individual book with specific title/author values

---
## Encapsulation

- Bundles data + methods into a class and **restricts access** to internal details
- Protects object state by exposing only what’s necessary
- Achieved through **access modifiers**:
    - **private** → internal use only
    - **protected** → accessible to class + subclasses
    - **public** → accessible everywhere
- Example:
    - BankAccount class hides internal balance (private)
    - Provides public deposit() and withdraw() methods
    - Optional: protected fields for subclasses like SavingsAccount

---
## Abstraction

- Defines **what** an object does, not **how** it does it
- Simplifies complexity; hides internal implementation
- Implemented through:
    - **Abstract classes** → mix of implemented + abstract methods
    - **Interfaces** → “contract” of required methods, no implementation
- Example:
    - Abstract base class defines deposit() / withdrawal() interfaces
    - CheckingAccount and SavingsAccount implement specifics
    - External code only needs to trust the interface

---
## Encapsulation + Abstraction Together

- Encapsulation → protects internal state via controlled access
- Abstraction → exposes only necessary behaviour through clear contracts
- Combined effect:
    - Secure code
    - Maintainable structure
    - Predictable interactions
    - Reduced complexity for developers
