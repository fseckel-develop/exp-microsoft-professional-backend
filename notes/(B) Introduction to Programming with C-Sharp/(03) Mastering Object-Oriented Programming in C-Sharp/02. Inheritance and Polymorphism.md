## Inheritance — Reusing and Extending Code

- Inheritance allows a **derived class** to be created from an existing **base class**
- Derived class receives all base-class attributes and methods automatically
- Developers can **add new features**, **extend** functionality, or **modify** inherited behavior
- Changes in derived classes are isolated and do **not** affect the base class
- Creates a **hierarchical, tree-like structure** that organizes code
- Analogy: parent → child; shared traits + unique characteristics

---
## Benefits of Inheritance

- **Code reuse** → no need to duplicate functionality
- **Extension** → add new methods/properties in derived classes
- **Customisation** → modify inherited behaviour only where needed
- **Organization** → natural hierarchy (e.g., Bird → Eagle / Parrot)

- **Example**:
	- Base class: Bird (declared with an access modifier + "class" keyword)
	    - Shared methods: fly()
	- Derived classes: (declared with "ClassName" : "BaseClassName")
	    - Eagle → adds hunt()
	    - Parrot → adds speak()

---
## Polymorphism — One Interface, Many Behaviours

- A single method name can perform **different actions** depending on the object
- Achieved primarily through **method overriding** in derived classes
- Allows developers to rely on a shared interface while enabling flexibility    

#### How Polymorphism Works

- Base class provides default method implementations
- Derived class may:
    - Use inherited method as-is
    - **Override** a method to change behaviour
    - Add completely new methods and properties

- **Example**:
	- Base class: CoffeeMaker
	    - Methods: water(), beans()
	- Derived class: Drip
	    - Uses inherited water() and beans()
	    - Adds vessel()
	- Derived class: SuperAutomatic
	    - Uses water()
	    - **Overrides beans()** to include bean-grinding functionality

---
## Method Overriding

- Derived class redefines a base-class method
- Enables context-specific behaviour
- Encourages reuse **without losing flexibility**
- Allows refined or completely replaced behaviour in child classes
- In C#:
	- Base class marks a method as "virtual"
	- Derived class overrides it using "override"

---
## Combined Impact: Flexible, Reusable OOP Design

- **Inheritance** → foundational structure + code reuse
- **Polymorphism** → adaptable behaviour per object type
- Together, they enable:
    - Cleaner architecture
    - Maintainable code
    - Reduced redundancy
    - Flexibility for future extensions
