## What Middleware Is

- Middleware is code that runs **before, after, or alongside** your route handlers
- It allows logic to be executed **across all routes or selected routes**, 
  rather than inside a single endpoint
- ASP.NET Core already uses middleware behind the scenes for 
  routing, authentication, error handling, and more
- Route handlers define _what an endpoint does_, 
  but middleware defines _what every request goes through_

---
## Starting from an Empty Web App

- A default empty app includes one GET route that returns “Hello World”
- That route handler responds only to /
- To run logic **outside** of that route, middleware is required

---
## Built-in Middleware

- Built-in middleware like **HTTP Logging** can be added using app.UseHttpLogging()
- It logs details about each request (path, status code, content type, timestamps, etc.)
- Requires enabling a service with builder.Services.AddHttpLogging()
- Needs log-level configuration inside appsettings.json
- Once enabled, it logs request/response information for **every route**

---
## Creating Your Own Middleware

- Custom middleware is added using app.Use(...)
- Middleware receives two parameters:
    - context → information about the current HTTP request
    - next → a function that triggers the next middleware or route handler
- Terminal middleware occurs when you **don’t** call next(), stopping the pipeline
- Most middleware:
    - Runs logic **before** calling next()
    - Runs additional logic **after** next() completes
- This creates a **before → next → after** chain, stacked in order

---
## Understanding Middleware Order

- Ordering is critically important
- Middleware added earlier runs its **before-logic** first
- After the request reaches a route handler, the response travels back upward
- Therefore:
    - “Before” logic executes **top to bottom**
    - “After” logic executes **bottom to top**
- Changing the order of app.Use(...) lines changes the entire pipeline flow

---
## Behind-the-Scenes Middleware in ASP.NET Core

- ASP.NET Core injects several middleware components automatically:
    - UseRouting → determines which route matches
    - UseAuthentication → only added if auth services exist
    - UseAuthorization → also conditional based on services
    - UseExceptionHandler → active in development for detailed error pages
    - UseEndpoints → executes the mapped route handlers
- These run **around** your custom middleware unless you override placement
- You typically do not need to re-add them manually

---
## Why Middleware Matters

- Enables application-wide behaviours such as 
  authentication, logging, caching, exception handling, or request validation
- Provides full control of the request and response lifecycle
- Helps build scalable, maintainable, and robust APIs
