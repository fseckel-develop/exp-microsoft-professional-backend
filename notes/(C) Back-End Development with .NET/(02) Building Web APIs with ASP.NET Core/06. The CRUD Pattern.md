## What CRUD Means

- CRUD stands for **Create, Read, Update, Delete**, the four essential operations used to manipulate data
- These actions apply to “records,” which can represent anything: blog posts, medical files, flights, social posts, etc.
- CRUD APIs form the backbone of nearly all data-driven systems on the internet

---
## Data Setup for the Example

- A simple Blog class with required Title and Body
- A temporary in-memory list stores example blog posts
- In real-world applications, CRUD logic targets a **database**, not an in-memory list
- The example focuses on teaching API mechanics rather than persistence

---
## Create Operation (C)

- **Create new blog**: POST /blogs
- The handler receives a Blog object from the request body
- Adds the blog to the list with blogs.Add(blog)
- Returns 201 Created with Results.Created() and a link to the new resource
- Requires a JSON body (e.g., { "title": "...", "body": "..." })

---
## Read Operations (R)

- **Read all**: GET /blogs returns the full list of blog posts
- **Read one**: GET /blogs/{id} returns a single blog based on index
- Route parameters are used to pass the ID into the handler
- Includes validation to handle invalid IDs (returns 404 Not Found)
- Uses Results.Ok() for proper HTTP responses

---
## Update Operation (U)

- **Update blog**: PUT /blogs/{id}
- Accepts both the ID and a new Blog object
- Validates ID before updating
- Performs replacement with blogs[id] = blog
- Returns updated data with Results.Ok(blog)

---
## Delete Operation (D)

- **Delete blog**: DELETE /blogs/{id}
- Validates index before deletion
- Uses blogs.RemoveAt(id) to remove the record
- Returns 204 No Content on success
- Demonstrates the importance of checking whether a record exists

---
## Completing the CRUD Cycle

- After implementing **Create**, **Read**, **Update**, and **Delete**, you have a fully functional CRUD API
- This same structure applies to any type of data: posts, users, inventory, transactions, etc.
- CRUD forms the foundation of most backend systems — 
  once you understand this pattern, you can construct almost any data-driven API
