## Dependencies and Services

- A dependency is anything a class/function relies on to work
- When the dependency is an object performing useful work, it is called a service
- Manually instantiating services inside route handlers causes tight coupling
- Tight coupling makes change difficult, duplicates logic, and hurts performance

---
## Why Tight Coupling Is a Problem

- Every route creating its own service instance wastes resources
- Changing one service implementation forces updates across many places
- Many instances created per request degrade performance
- Difficult to test because dependencies are hard-coded

---
## Loose Coupling

- Components depend on _abstractions_, not concrete implementations
- Interfaces act as contracts (e.g., IAuthService)
- Code only cares about _what_ a service does, not _how_ it does it
- Improves flexibility, maintainability, and adaptability

---
## What Dependency Injection (DI) Does

- Services are provided **from the outside**, not created internally
- DI container decides _when_ and _how_ services are constructed
- Route handlers simply “accept” the services they need
- Makes swapping implementations easy (database auth → API auth)

---
## Benefits of Dependency Injection

- Externalised service creation allows central configuration
- Reduces repeated instantiation, improving performance
- Easier to maintain — change in one place updates all routes
- Enables easy swapping of service implementations
- Ideal for testing because services can be replaced with mocks

---
## Creating Services in ASP.NET Core

- **Defining a Service**:
	- Create an interface (e.g., IMyService) describing required methods
	- Implement a class (MyService) that fulfills the interface
	- Services are just objects performing utility tasks for the application
	- Example service generates a unique ID for demonstration

- **Registering Services**
	- Services registered via builder.Services before app build
	- Registration requires interface + implementation
	- Example: AddSingleton<IMyService, MyService>()
	- Registration happens before app = builder.Build()

- **Injecting Services into Handlers**:
	- Route handlers receive services automatically by listing them as parameters
	- Example: (IMyService myService) => { ... }
	- Middleware retrieves services using context.RequestServices.GetRequiredService()

---
## Service Lifetimes

- **Singleton**:
	- Only one instance exists for the _entire_ application lifetime
	- All requests and middleware share the same instance
	- Good for expensive, shared resources (e.g., DB connection pools)

- **Scoped**:
	- One instance per HTTP request
	- Middleware and handlers share the same instance **within that request**
	- A new instance is created for each new request
	- Ideal for per-request operations (e.g., unit-of-work patterns)

- **Transient**:
	- New instance created _every time_ the service is requested
	- Middleware and handlers each get separate instances
	- Useful for lightweight, stateless services

---
## Dependency Injection in Middleware

- Middleware retrieves services via the request’s service provider
- Example: context.RequestServices.GetRequiredService< IMyService >()
- Multiple middleware components using scoped or transient services
  illustrate lifetime behaviour
- Order matters—middleware runs in pipeline sequence

## Unit Testing with DI and Mocking

- **Why Unit Testing Matters**:
	- Tests each component in isolation
	- Catches bugs early before integration
	- Improves reliability and code quality

- **Why Mocking Is Needed**:
	- Real services (DB, APIs) are too slow, unstable, or unavailable for tests
	- Mocks simulate external behaviour to isolate the component under test
	- Makes tests predictable and fast

- **Mocking Process**:
	- Install a mocking framework (e.g., Moq)
	- Configure mock to return specific data
	- Inject mock service instead of real implementation
	- Run component logic using controlled inputs
	- Use assertions to verify expected outputs

- **Example: “Add to Cart” Testing**:
	- Mock product database returns predefined product info
	- Call Add-to-Cart with mock service
	- Verify cart contains correct product ID, price, and quantity
	- Test runner executes tests and reports pass/fail
