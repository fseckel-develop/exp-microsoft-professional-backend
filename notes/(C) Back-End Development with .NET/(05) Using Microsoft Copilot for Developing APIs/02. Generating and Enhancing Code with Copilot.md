## Generation of API Code

- Copilot can be used via desktop app, web (copilot.microsoft.com), or sidebar chat; 
  all provide a chat-based interface
- Vague prompts produce incorrect results (e.g., Node.js instead of .NET)
- Clear **context is critical**: specify language, framework, API style, data storage, constraints

- **Well-scoped prompts can generate**:
    - Complete minimal API CRUD endpoints in ASP.NET Core
    - In-memory data models (no database)
    - Sample test files (e.g., .http requests for VS Code)

- **Copilot is especially effective for**:
    - Scaffolding APIs quickly
    - Generating test requests
    - Explaining unfamiliar code (LINQ, ternaries, etc.)

- **Always review generated code before running it**
    - Verify logic, security, and correctness
    - Expect small fixes (e.g., required properties)

- **Best use case**:
    - Small, well-defined tasks and prototypes
    - Larger projects benefit more from IDE-integrated Copilot

## Enhancement of API Code

- **Refactoring Data Storage**:
	- Lists caused fragile IDs and bounds issues
	- Copilot suggested a dictionary for stable, explicit IDs
	- Improved safety and flexibility without changing API endpoints

- **Applying AI-Generated Code Safely**:
	- Always review output for missing or altered code
	- Watch for hallucinations (removed routes, setup code)
	- Use version control before large changes

- **Preserving the API Contract**:
	- Internal changes did not affect existing requests
	- Clients continued working without updates

- **Making Focused Changes**:
	- Ask Copilot to modify only specific routes or functions
	- Reduces risk of unintended side effects

- **Adding Search Functionality**:
	- Added query parameter to filter blogs by title/body
	- Used LINQ over dictionary values
	- Verified behaviour with existing HTTP tests

- **Key Takeaways**:
	- Give Copilot clear context and constraints
	- Prefer small, isolated prompts
	- Treat AI output as a draft, not final code
