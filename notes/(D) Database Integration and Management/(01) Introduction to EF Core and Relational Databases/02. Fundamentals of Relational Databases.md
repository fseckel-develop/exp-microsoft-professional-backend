## Overview

- Relational databases provide a **structured and efficient way** to 
  store, search, update, and analyse large amounts of data.
- Widely used in backend development due to their **organisation, scalability, and reliability**.
- Data is managed using tables, relationships, normalisation, and constraints to ensure consistency and integrity.

---
## Core Structure of Relational Databases

- **Tables**:
	- Tables are the **building blocks** of relational databases.
	- Each table represents a specific subject (e.g., Employees, Departments, Books).
	- Data is organised in a grid format of rows and columns.

- **Columns (Attributes)**:
	- Columns define the **type of data** stored in a table.
	- Each column stores a single kind of data (e.g., names, salaries, IDs).
	- Data types are consistent within a column.

- **Rows (Records)**:
	- Rows represent **individual records**.
	- Each row contains a complete set of data across all columns for one entity.
	- Example: One row represents one employee or one book.

---
## Schemas and Database Organization

- Schemas act as a **blueprint** for the database.
- They define:
	- Table structures
	- Column definitions
	- Relationships between tables
- Schemas ensure consistency and logical organisation.

---
## Keys and Table Relationships

- **Primary Keys (PK)**:
	- A primary key uniquely identifies each row in a table.
	- Guarantees that no duplicate records exist.
	- Example: `EmployeeID`, `BookID`.

- **Foreign Keys (FK)**:
	- A foreign key references a primary key in another table.
	- Establishes relationships between tables.
	- Maintains data integrity by enforcing valid links.

---
## Types of Database Relationships

- **One-to-One**:
	- One record in a table corresponds to exactly one record in another table.
	- Example: A book linked to one unique special edition.

- **One-to-Many**:
	- One record relates to multiple records in another table.
	- Example: One author can write many books.

- **Many-to-Many**:
	- Multiple records in one table relate to multiple records in another.
	- Implemented using a **junction table**.
	- Example:
		- Authors ↔ Books
		- Junction table (e.g., `AuthorBooks`) stores author–book associations.

---
## Normalisation

- **Purpose of Normalisation**:
	- Reduces data redundancy.
	- Improves data integrity.
	- Ensures efficient data storage and maintenance.

- **First Normal Form (1NF)**:
	- Each column contains **atomic (indivisible) values**.
	- No repeating groups or multi-valued fields.

- **Second Normal Form (2NF)**:
	- Eliminates partial dependencies.
	- Non-primary key attributes depend entirely on the primary key.

- **Third Normal Form (3NF)**:
	- Removes transitive dependencies.
	- All data in a table depends **only on the primary key**.
	- Related data is moved into separate tables when appropriate.

---
## Constraints

- **Purpose of Constraints:**
	- Enforce rules to keep data **accurate, valid, and consistent**.

- **Common Constraint Types**:
	- **NOT NULL**:
		- Prevents empty values in critical fields.
	
	- **UNIQUE**
		- Ensures all values in a column are distinct (e.g., ISBN).
	
	- **CHECK**
		- Enforces conditions on data values (e.g., publication year > 1500).
	
	- **DEFAULT**
		- Assigns a default value when none is provided (e.g., default price).

---
## Key Benefits of Relational Databases

- Organised and logical data structure
- Reduced redundancy through normalisation
- Strong data integrity via keys and constraints
- Scalable and maintainable as data grows
- Efficient querying and retrieval

---
## Key Takeaway

- Relational databases rely on **tables, keys, relationships, normalisation, and constraints**.
- These fundamentals create a scalable, consistent, reliable foundation for backend systems.
- Mastering this is essential for designing robust and maintainable data-driven applications.
