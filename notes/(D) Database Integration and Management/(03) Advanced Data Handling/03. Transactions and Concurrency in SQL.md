## Introduction to SQL Transactions

- SQL transactions ensure database changes occur accurately and reliably  
- A transaction is a sequence of operations treated as one logical unit of work
- All operations in a transaction must succeed together or fail together  
- Transactions prevent partial updates and protect data integrity  

### ACID Properties of Transactions

- **Atomicity**
	  - Ensures a transaction is all-or-nothing  
	  - Partial updates are never applied  
	  - If any step fails, the entire transaction is rolled back  

- **Consistency**
	  - Guarantees the database moves from one valid state to another  
	  - All rules, constraints, and integrity checks must be satisfied  
	  - Prevents invalid data states  

- **Isolation**
	  - Ensures transactions run independently of one another  
	  - Changes made by one transaction are not visible to others until committed  
	  - Prevents interference when transactions overlap  

- **Durability**
	  - Ensures committed changes are permanently saved  
	  - Data remains intact even after crashes or system failures  

### Transaction Control Commands

- **BEGIN TRANSACTION**
	  - Marks the start of a transaction  
	  - Groups subsequent operations into a single unit of work  

- **COMMIT**
	  - Finalises the transaction  
	  - Permanently saves all changes to the database  

- **ROLLBACK**
	  - Cancels the transaction  
	  - Reverses all changes made since the transaction began  

### Importance of Transactions

- Transactions are critical for operations requiring strict accuracy  
- Common use cases include financial transfers and inventory updates  
- They ensure databases remain reliable during complex changes

## Concurrency Control Overview

- Concurrency control manages simultaneous access to shared data  
- Prevents conflicts when multiple transactions run at the same time  
- Ensures accuracy and consistency in multi-user environments  

### Common Concurrency Issues

- **Dirty Reads**
	  - Occur when a transaction reads uncommitted data  
	  - Can result in invalid or misleading information  

- **Non-Repeatable Reads**
	  - Happen when data changes between multiple reads in a transaction  
	  - Leads to inconsistent results  

- **Phantom Reads**
	  - Occur when new rows appear in repeated queries within a transaction  
	  - Causes shifting result sets  

### Isolation Levels in SQL

- **Read Uncommitted**
	  - Allows reading uncommitted data  
	  - Fast but prone to all concurrency issues  

- **Read Committed**
	  - Prevents dirty reads  
	  - Allows non-repeatable and phantom reads  

- **Repeatable Read**
	  - Prevents dirty and non-repeatable reads  
	  - Ensures stable data during a transaction  

- **Serialisable**
	  - Highest isolation level  
	  - Prevents all concurrency issues  
	  - Most restrictive and can impact performance  

### Locking Mechanisms

- **Shared Locks**
	  - Allow multiple transactions to read data  
	  - Prevent modifications until released  

- **Exclusive Locks**
	  - Prevent other transactions from reading or writing data  
	  - Used for critical updates  

- **Update Locks**
	  - Signal intent to modify data  
	  - Prevent multiple transactions from updating the same data simultaneously

## Summary

- SQL transactions ensure reliable, consistent, and accurate database changes  
- ACID properties form the foundation of transaction safety  
- Concurrency control prevents conflicts in multi-user systems  
- Isolation levels and locking balance performance with data integrity  
- Together, these mechanisms ensure databases remain trustworthy under heavy use
