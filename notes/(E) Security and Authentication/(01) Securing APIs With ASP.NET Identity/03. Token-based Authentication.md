## Overview

- Token-based authentication allows users to stay authenticated without repeated logins
- Uses reusable tokens instead of server-side sessions
- Commonly applied to:
	- APIs
	- Distributed systems
	- Front-ends that cannot rely on cookies
- Balances convenience, scalability, and security

---
## Core Concepts

### Tokens

- Tokens are small pieces of data used as credentials
- Represent a user’s identity and access rights
- Typically:
	- Encrypted or digitally signed
	- Protected against tampering
- Sent with each request to verify authenticity

### Token-Based Authentication

- User logs in once with credentials
- Server issues a token after successful authentication
- Token is included in all subsequent requests
- Eliminates the need to resend credentials repeatedly
- Common example:
	- Social media platforms maintaining login state

---
## JSON Web Tokens (JWT)

### JWT Characteristics

- JWTs store user information as a compact JSON object
- Common contents:
	- User ID
	- Permissions or roles
- Signed to ensure:
	- Data integrity
	- Authenticity
- Widely used for authentication and authorisation

### JWT Usage Example

- User logs into an application (e.g., student platform)
- Token defines allowed actions (courses, discussions, submissions)
- Access is granted or denied based on token data
- Restricted areas are automatically blocked

---
## Stateless Authentication

### Stateless Model

- Server does not store session data
- Each request includes a token
- Server independently validates each request
- Improves:
	- Performance
	- Scalability
	- Fault tolerance

### Stateless Authentication Example

- Online banking actions validate tokens per request
- No session memory required on the server
- Ensures secure, efficient request handling

---
## Securing APIs with Tokens

- Tokens protect API endpoints from unauthorised access
- API validates:
	- Token authenticity
	- Token permissions
- Example:
	- Fitness app restricts workout data to authenticated users
- Prevents exposure of sensitive data

---
## Token-Based Authentication Workflow

### Authentication Flow

- User submits credentials
- Server validates credentials
- Server issues a token (often a JWT)
- Token contains identity and permission data

### Request Authorization

- Client includes token with each request
- Server validates token:
	- Signature
	- Expiration
	- Permissions
- Access is granted or denied based on validation

### Token Storage

- Tokens are stored client-side:
	- Cookies (more secure in browser environments)
	- Local storage (used when cookies are unavailable)
- Storage choice depends on client capabilities

---
## Cookies vs Tokens

### Cookies

- Automatically handled by browsers
- Default and preferred option when supported
- Can be invalidated on logout
- Ideal for traditional web applications

### Session Tokens

- Manually managed by the client
- Sent explicitly via headers
- Commonly used when cookies are not supported
- Require custom client-side handling

---
### Bearer Tokens

- Tokens are sent using the Authorisation header
- Format:
	- `Authorization: Bearer <token>`
- Server validates token before allowing access
- Incorrect or modified tokens result in denial

---
## Token Expiration and Security

### Token Lifetime

- Tokens are short-lived for security reasons
- If compromised, exposure time is limited
- Cannot be revoked like cookies
- Logout does not invalidate existing tokens

### Refresh Tokens

- Used to request new access tokens
- Stored securely on the client
- Sent in request payload (not headers)
- Allow seamless re-authentication without login
- Refreshing tokens:
	- Does not invalidate older tokens
	- Issues new tokens with updated expiration

### Token Refresh Flow

- Client sends refresh token to refresh endpoint
- Identity validates refresh token
- Server returns a new access token
- Client replaces expired token
- Process repeats as needed

---
## Summary

- Token-based authentication enables stateless, scalable security
- Ideal for APIs and non-browser clients
- Requires careful handling of:
	- Token storage
	- Expiration
	- Refresh mechanisms
- Short-lived tokens and refresh tokens mitigate security risks
- Complements cookie-based authentication depending on environment
