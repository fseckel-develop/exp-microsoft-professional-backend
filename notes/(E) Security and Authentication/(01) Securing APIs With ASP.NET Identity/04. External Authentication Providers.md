## Overview

- External Authentication Providers allow users to sign in using trusted third-party services
- Common providers include:
	- Microsoft
	- Google
	- Facebook
- Integrated into ASP.NET applications via secure authentication protocols
- Improve usability while strengthening security

---
## Definition and Purpose

- External Authentication Providers are third-party services that handle user authentication
- They verify user identities using established security standards
- Applications rely on providers instead of storing user passwords
- Enables developers to focus on application logic rather than credential management

---
## Benefits of External Authentication

### Simplified Login Experience

- Users log in using existing accounts
- Eliminates the need to create and manage new passwords
- Example:
	- “Sign in with Microsoft” or “Sign in with Google”

### Enhanced Security

- Providers manage password storage and protection
- Often include advanced security features:
	- Two-factor authentication
	- Device or email verification
- Reduces risk of credential leaks in the application

### Increased User Trust

- Users are more comfortable using well-known platforms
- Trusted providers improve adoption and confidence
- Reduces friction during onboarding

---
## Authentication Protocols

### OAuth 2.0

- Authorisation protocol used by external providers
- Grants limited access using secure tokens
- Tokens:
	- Represent user authorisation
	- Avoid sharing passwords with applications
- Example:
	- Token issued when logging in with a Microsoft account

### OpenID Connect (OIDC)

- Identity layer built on top of OAuth 2.0
- Confirms who the user is
- Supplies identity information as claims
- Ensures accurate identity verification
- OAuth 2.0 controls access
- OIDC verifies identity

---
## Integration in ASP.NET Applications

### Authentication Setup

- ASP.NET supports external authentication through middleware
- Requires:
	- Authentication configuration
	- Authorisation services
- Cookies are commonly used as the default authentication scheme
- External provider acts as the challenge scheme

### Google Authentication Integration

- Requires installing Google authentication package
- Uses OAuth 2.0 for login flow
- Configuration includes:
	- Client ID
	- Client Secret
- Secrets must be stored securely:
	- Environment variables
	- App settings excluded from source control

### Application Flow

- Public routes are accessible without authentication
- Secured routes trigger external provider login
- User is redirected to provider (e.g., Google)
- After successful login:
	- User is redirected back
	- Access is granted to secured routes

---
## Provider Credential Setup (Google Example)

### OAuth Credentials Creation

- Created in Google Cloud Console
- Steps include:
	- Configuring consent screen
	- Creating OAuth 2.0 Client ID
- Application type:
	- Web application

### Redirect URIs

- Required for authentication callbacks
- Must match application URLs
- Typically include:
	- HTTP and HTTPS variants
	- Default provider-specific callback paths

### Credential Usage

- Client ID and Client Secret are added to application configuration
- Used by authentication middleware
- Enables secure communication with the provider

---
## Summary

- External Authentication Providers improve login experience and security
- OAuth 2.0 and OpenID Connect work together to manage access and identity
- ASP.NET supports seamless integration with providers like Google
- Proper configuration and secret management are critical
- Enables users to authenticate using trusted, familiar platforms
