## Introduction

- Apps frequently exchange sensitive data such as medical records or financial information
- Each data exchange introduces potential security risks
- JSON Web Tokens help secure API endpoints by ensuring only trusted users gain access

---
## What Is an API Endpoint?

- An API endpoint is a specific URL exposed by a server
- Clients send requests to these URLs to:
	- Retrieve data
	- Submit information
	- Perform actions on the server
- Securing endpoints is critical to prevent unauthorised access

---
## JWT Validation as a Security Checkpoint

- JWT validation ensures that a token is:
	- Authentic
	- Untampered
	- Still valid
- Only requests with valid tokens are allowed to access protected endpoints

---
## Key Steps in JWT Validation

- Signature verification:
	- Server checks the token’s digital signature
	- Ensures the token has not been altered
- Expiration validation:
	- Tokens include an expiration timestamp
	- Expired tokens are rejected
	- Users must re-authenticate to obtain a new token

---
## Role of JWT Middleware in ASP.NET Core

- JWT middleware acts as an automated security gate
- It validates tokens on every incoming request
- Middleware checks:
	- Token authenticity
	- Token expiration
- Invalid tokens are blocked before reaching API routes
- Valid tokens are passed through to secured endpoints

---
## Adding Role-Based Access Control (RBAC)

- RBAC adds an extra layer of security on top of JWT validation
- Access is determined by the user’s role
- Roles define what actions a user can perform
- Ensures users only access data and features they are authorised for

---
## Adding Roles to JWTs

Roles are added directly to the JWT payload:

- Key: role  
- Value: admin  

Once included:

- Middleware detects the role claim  
- Role-based policies can validate access  

The role key is case-sensitive and must match exactly.

---
## Adding Claims to JWTs

Claims follow a key-value structure:

- Key: department  
- Value: IT  

Claim-based policies validate both the key and value.

Important notes:

- Claims are case-sensitive  
- Policy requirements must match token values exactly  

---
## Combining Roles and Claims

A single JWT can contain:

- Multiple roles
- Multiple claims

This allows the same token to be used across different protected routes, each enforcing different authorisation policies.

ASP.NET Core simply checks whether:

- The required role exists  
- The required claim key and value exist  

---
## Example: Securing Medical API Endpoints

- User authentication:
	- User logs in
	- Server generates a JWT for that user
- Token contents:
	- Claims defining role and permissions
	- Examples:
		- Doctor role
			- Can edit patient records
		- Patient role
			- Can only view personal information
- Request handling:
	- JWT middleware validates the token on each request
	- RBAC checks permissions based on role claims
	- Unauthorised access is immediately denied

---
## Security Benefits of JWT-Based Endpoint Protection

- Ensures every request is authenticated
- Enforces fine-grained access control through roles
- Protects sensitive data from unauthorised users
- Reduces manual validation logic through middleware

---
## Conclusion

- Securing API endpoints with JWTs ensures safe data access
- Token validation protects against tampering and expired access
- Middleware automates security checks for every request
- Combining JWTs with role-based access control provides robust protection
- This approach is essential for applications handling sensitive information
