## Overview

- Redis is a distributed, in-memory data store
	- Used as a cache, database, and message broker
	- Optimised for fast data access and scalability
- In .NET applications
	- Redis is accessed via the `IDistributedCache` interface
	- Enables shared caching across multiple servers
- Core benefits
	- Reduced database load
	- Faster response times
	- Consistent data access in distributed systems

---
## Redis and IDistributedCache Basics

- Redis
	- Open-source, in-memory data store
	- Centralised cache accessible by all application servers
- IDistributedCache
	- .NET interface for distributed caching solutions
	- Simplifies storing and retrieving cached data
	- Common use cases
		- User preferences
		- Session data
		- Frequently accessed application data

---
## Setting Up Redis in .NET

- Redis installation options
	- Local installation (Memurai, Homebrew, APT)
	- Docker-based setup
- Application configuration
	- Add Redis connection settings in `appsettings.json`
	- Register Redis in `Program.cs` using `AddStackExchangeRedisCache`
- Caching interface
	- Use `IDistributedCache` to interact with Redis
	- Supports asynchronous cache operations

---
## Storing Data in Redis

- Purpose
	- Cache frequently accessed data to reduce database queries
- Key-value storage
	- Keys uniquely identify cached items
	- Values stored as byte arrays (e.g., UTF-8 encoded strings)
- Example use case
	- User preference data
		- Key: `user:123:category`
		- Value: `electronics`
- Cache configuration
	- Use `SetAsync` to store data
	- Define expiration using `DistributedCacheEntryOptions`
		- Example: absolute expiration after 1 hour

---
## Retrieving Cached Data

- Data access
	- Use `GetAsync` to retrieve values by key
- Cache hit
	- Data is found in Redis
	- Returned immediately without database access
- Cache miss
	- Data not found in Redis
	- Application retrieves data from the database
	- Data is stored back in Redis for future requests
- Fallback logic
	- Ensures application reliability when cache entries expire

---
## Cache Synchronization and Consistency

- Distributed system challenge
	- Multiple servers access the same cached data
- Centralized Redis instance
	- Ensures all servers retrieve the same, latest values
- Example scenario
	- User updates preferred category to `fashion`
		- Stored centrally in Redis
		- All servers retrieve updated value
- Result
	- Prevents stale or inconsistent data
	- Ensures uniform user experience across servers

---
## Monitoring Cache Performance

- Importance of monitoring
	- Ensures caching is effective
	- Identifies inefficiencies
- Cache hits
	- Data served directly from Redis
	- Indicates optimal cache usage
- Cache misses
	- Data fetched from database
	- May signal expired entries or misconfiguration
- Logging strategy
	- Log cache hits and misses
	- Include cache keys and fallback actions
- Optimization
	- Analyze logs to adjust expiration policies
	- Reduce unnecessary cache misses
