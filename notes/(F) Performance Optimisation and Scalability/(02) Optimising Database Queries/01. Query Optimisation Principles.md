## Overview

Query optimisation improves database performance by minimising resource consumption such as execution time, CPU usage, and memory load. Instead of storing data like caching does, query optimisation focuses on refining how the database retrieves data. Efficient query optimisation ensures applications remain responsive and scalable as demand increases.

---
## Fundamentals of Query Optimisation

Query optimisation is the process of improving how efficiently a database executes queries. It aims to reduce server load while delivering accurate results quickly.

Key characteristics of query optimisation include:
- Reducing execution time
- Minimising CPU, memory, and I/O usage
- Improving scalability under high traffic

Unlike caching, which temporarily stores data for faster access, query optimisation ensures the database retrieves data in the most efficient way possible every time a query runs.

---
## Query Optimisation Process

The query optimisation consists of several steps working together to improve performance:

### Query Analysis and Execution Plans

- Query execution plan is a detailed set of instructions describing how database retrieves data.
- Tools such as MySQL’s `EXPLAIN` or SQL Server’s **Actual Execution Plan** reveal:
	- Whether indexes are used
	- Whether tables are scanned fully
	- The order of operations
- Analysing execution plans helps identify inefficiencies and optimisation opportunities.

### Indexing

- Indexing creates data structures (e.g., B-trees or hash tables) that speed up data retrieval.
- Indexes act as shortcuts, allowing the database to locate rows without scanning entire tables.
- Example:
	- Indexing an email column allows fast lookup of a customer by email.
- Without indexes, the database must examine every row, which significantly increases execution time.

### Cost-Based Optimisation

- Databases use cost-based optimisers to evaluate multiple execution plans.
- Each plan is assessed based on resource usage such as CPU, memory, and I/O.
- The optimiser selects the plan with the lowest estimated cost.
- Example:
	- Choosing an indexed lookup instead of a full table scan.
	- Using precomputed results instead of recalculating data.

### JOIN Optimisation

- JOINs combine data from multiple tables and can be resource-intensive.
- Optimising joins reduces unnecessary computation.
- Common techniques include:
	- Filtering smaller datasets before joining
	- Choosing efficient join algorithms
- Efficient joins improve overall query performance in normalised relational databases.

## Execution Plans and Performance Impact

Query execution follows a structured sequence of steps:

### Parsing

- The database validates the query structure.
- Ensures syntax and required components (tables, columns, actions) are correct.
- Parsing errors prevent the query from proceeding.

### Cost Calculation

- The optimiser evaluates different execution paths.
- Determines whether to use indexes or scan tables.
- Selects the most efficient plan based on estimated cost.

### Execution

- The database retrieves the requested data using the chosen plan.
- Indexed plans retrieve only relevant rows quickly.
- Table scans process every row, which is slower and more resource-intensive.

---
## Why Query Optimisation Matters

- Reduces query execution time
- Lowers server load and infrastructure costs
- Improves application responsiveness under heavy traffic
- Ensures scalable and efficient database operations

---
## Conclusion

Optimising queries is essential for building fast and scalable applications. By analysing execution plans, leveraging indexing, applying cost-based optimisation, and refining joins, databases can process queries efficiently. Effective query optimisation minimises resource usage while maintaining high performance, benefiting both developers and end users.
