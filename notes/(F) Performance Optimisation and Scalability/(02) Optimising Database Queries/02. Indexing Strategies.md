---
## Indexing Fundamentals

- Index definition
	- Data structure that organizes data for faster retrieval
	- Comparable to a book index pointing directly to information
- Purpose of indexing
	- Reduces full table scans
	- Improves query execution speed on frequently queried columns
- Practical example
	- Indexing a country column enables fast retrieval of customers from Canada
	- Avoids scanning all rows in a customer table

---
## Types of Indexes

### Clustered Index

- Core characteristics
	- Determines the physical order of rows in a table
	- Typically created automatically on the primary key
- Performance impact
	- Highly efficient for range queries and sorted data access
- Analogy
	- Books arranged on a shelf by ISBN
- Best-use scenarios
	- Frequently queried ranges
	- Sorted or sequential data access

### Non-Clustered Index

- Core characteristics
	- Stores pointers to the actual data rows
	- Does not alter physical data order
- Performance impact
	- Fast lookups on non-primary-key columns
- Analogy
	- Textbook index pointing to page numbers
- Example
	- Indexing product names to enable quick searches

### Composite Index

- Core characteristics
	- Index built on multiple columns
- Performance impact
	- Improves performance for queries filtering on multiple fields
- Trade-off
	- Excessive composite indexes can degrade write performance

---
## Indexing Use Cases and Benefits

- Faster search operations
	- Efficient retrieval for frequent queries (e.g. orders by customer ID)
- Improved query optimisation
	- Database locates rows directly instead of scanning entire tables
- High-demand scenarios
	- Social media feeds
	- E-commerce product searches
	- Customer and account lookups

---
## Trade-Offs and Challenges of Indexing

### Write Performance Impact

- Index maintenance overhead
	- Inserts, updates, and deletes require index updates
- Performance concern
	- Write-heavy systems may slow down
- Optimisation strategy
	- Disable indexes during bulk inserts
	- Rebuild indexes after data loads

### Storage Overhead

- Additional storage usage
	- Each index adds extra data structures
- Scalability concern
	- Large databases with many indexes increase storage costs
- Example
	- News platforms indexing millions of articles

### Over-Indexing Risks

- Definition
	- Creating too many or unnecessary indexes
- Negative effects
	- Slower write operations
	- Increased maintenance complexity
- Real-world example
	- Video streaming platforms indexing every attribute, slowing updates

---
## Measuring Performance Improvements

- Execution time analysis
	- Compare query duration before and after indexing
	- Use tools like SET STATISTICS TIME ON
- Resource usage monitoring
	- Track CPU and memory consumption
- Query execution plans
	- Use EXPLAIN (e.g. in MySQL) to confirm index usage
- Performance testing practices
	- Verify indexes prevent full table scans
	- Benchmark queries with and without indexes
	- Monitor resource utilisation over time

---
## Common Pitfalls to Avoid

- Over-indexing
	- Too many indexes slow inserts, updates, and deletes
- Unused indexes
	- Consume storage without performance benefits
- Incorrect index selection
	- Using non-clustered indexes where clustered indexes are more suitable
- Lack of analysis
	- Failing to verify index usage via execution plans

---
## Best Practices for Indexing

- Target high-value columns
	- Focus on frequently queried or filtered fields
- Balance read and write performance
	- Avoid indexing rarely used attributes
- Monitor and refine indexes
	- Use query logs to identify high-impact columns
	- Remove unused or low-benefit indexes
- Continuous optimisation
	- Adjust indexing strategy as application usage evolves

---
## Conclusion

- Indexes are essential for fast and efficient SQL query execution
- Clustered, non-clustered, and composite indexes serve distinct purposes
- Improper indexing can harm performance through write slowdowns and storage overhead
- Measuring performance and avoiding common pitfalls is critical
- A balanced, well-monitored indexing strategy ensures scalability and responsiveness
