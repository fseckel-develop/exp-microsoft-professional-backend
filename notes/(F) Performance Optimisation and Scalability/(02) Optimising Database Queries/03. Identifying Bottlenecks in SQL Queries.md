## Overview

- Databases store large volumes of data, but inefficient queries hinder retrieval
- SQL optimisation focuses on identifying and removing performance bottlenecks
- Query performance metrics help diagnose inefficiencies in complex queries

---
## Full Table Scans

- Occur when the database checks every row in a table
- Common causes:
	- Missing indexes
	- Poorly structured `WHERE` clauses
- Example issue:
	- Using `SELECT *` retrieves all columns, including unnecessary ones
	- Acceptable for small tables but inefficient for large datasets
- Optimisation approaches:
	- Select only required columns
	- Apply `WHERE` clauses to filter rows early
	- Create indexes on frequently filtered columns

---
## Joins on Large Tables

- Become expensive when tables grow in size
- Performance issue arises when:
	- Join columns are not indexed
- Resulting behaviour:
	- Full scans of both tables
	- Row-by-row comparisons
- Optimisation approaches:
	- Create indexes on join columns (e.g. `CUSTOMER_ID`)
	- Allow faster row matching and reduced scan operations

---
## Poor Use of Temporary Tables

- Temporary tables store intermediate query results
- Performance drawbacks:
	- Increased memory consumption
	- Additional execution overhead
- Example issue:
	- Creating a temporary table before applying aggregation
- Optimisation approaches:
	- Combine filtering and aggregation into a single query
	- Eliminate intermediate storage when possible

---
## Unnecessary Nested Subqueries

- Subqueries execute repeatedly for each row
- Increase:
	- Query complexity
	- Execution time
- Example issue:
	- Checking existence of values via subqueries for each row
- Optimisation approaches:
	- Rewrite subqueries as JOINs
	- Perform filtering and matching in a single execution step

---
## Lack of Performance Analysis

- Performance issues remain hidden without analysis tools
- `EXPLAIN PLAN` provides:
	- Step-by-step execution details
	- Insight into index usage and scan methods
- Key indicators:
	- `TABLE ACCESS FULL`
		- Indicates full table scan
		- Signals missing or unused indexes
	- `INDEX SCAN`
		- Indicates efficient index-based retrieval
		- Reduces rows scanned and improves speed

---
## Key Takeaways

- Common SQL bottlenecks include:
	- Full table scans
	- Unindexed joins
	- Temporary tables
	- Nested subqueries
- Using indexes, simplifying query structure, and analysing execution plans:
	- Improves performance
	- Reduces resource usage
	- Enables scalable database operations
