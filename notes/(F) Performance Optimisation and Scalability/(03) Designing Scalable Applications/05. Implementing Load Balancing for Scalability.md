## Introduction

-	Load balancing is a core technique for scalable and reliable systems  
-	It distributes incoming traffic across multiple servers  
-	Prevents overload, crashes, and performance degradation  
-	Ensures responsiveness during traffic spikes (e.g. flash sales, product launches)  

---
## Role and Purpose of Load Balancing

-	Primary objectives:
	-	Distribute traffic evenly across servers
	-	Prevent single points of failure
	-	Maintain high availability and consistent performance

-	User experience impact:
	-	Faster response times
	-	Reduced downtime
	-	Smoother interactions under heavy load

---
## Core Load Balancing Concepts

### Load Balancer

-	Acts as a traffic director between users and backend servers  
-	Routes requests to available and healthy servers  
-	Can be software-based (e.g. NGINX) or cloud-managed  

### Load Balancing Algorithms

-	Round-robin:
	-	Cycles through servers sequentially
	-	Ensures even workload distribution
-	Least connections:
	-	Sends traffic to the server with the fewest active connections
-	Weighted round-robin:
	-	Distributes traffic based on server capacity
-	IP hashing:
	-	Routes users based on IP address for consistent server assignment

---
## Health Checks and Reliability

### Health Checks

-	Regular tests to verify server availability  
-	Detect unresponsive or failed servers  
-	Automatically remove unhealthy servers from rotation  

### Fault Handling

-	Traffic is redirected to healthy servers when failures occur  
-	Supports maintenance, restarts, and unexpected outages  
-	Improves overall system resilience  

---
## Sticky Sessions (Session Persistence)

-	Keeps a user connected to the same backend server  
-	Preserves session context (e.g. shopping carts, preferences)  
-	Useful for stateful or personalized interactions  
-	Reduces repeated data retrieval from shared storage  

---
## Practical Implementation with NGINX

### Backend Setup

-	Multiple backend servers running identical applications  
-	Each server listens on a different port or machine  
-	Example:
	-	Server 1 → Port 5001
	-	Server 2 → Port 5002

### NGINX Configuration

-	Defines upstream backend servers  
-	Listens on a single public endpoint (e.g. port 80)  
-	Distributes requests using round-robin by default  

### Traffic Distribution Behaviour

-	Requests alternate between backend servers  
-	If one server fails:
	-	NGINX detects failure via health checks
	-	Traffic is routed only to active servers  

---
## Traffic Distribution Strategies

-	Round-robin:
	-	Simple and effective for similar server capacities
-	Least connections:
	-	Better for uneven workloads
-	Weighted strategies:
	-	Optimized for mixed-capacity infrastructure
-	IP-based routing:
	-	Ensures session consistency when required  

---

## Troubleshooting Load Balancer Issues

### Common Problems

-	High latency:
	-	Backend overload
	-	Network congestion
-	Uneven traffic distribution:
	-	Misconfigured algorithms
	-	Incorrect server weights
-	Frequent server failures:
	-	Poor health check configuration
	-	Unstable backend services
-	Session persistence issues:
	-	Misconfigured sticky sessions

### Mitigation Strategies

-	Scale backend servers horizontally  
-	Adjust load-balancing algorithms and weights  
-	Improve health check intervals and thresholds  
-	Ensure proper session handling configuration
