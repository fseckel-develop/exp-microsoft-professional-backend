## Introduction

Database performance heavily depends on how efficiently SQL queries retrieve and process data. Even small inefficiencies—such as retrieving unnecessary columns or using unoptimized joins—can accumulate over time, causing slow response times, excessive resource usage, and scalability issues.

In this lesson, we explore:

- Common SQL query inefficiencies  
- Techniques to optimize query performance  
- How Microsoft Copilot assists in detecting and refactoring inefficient queries  

---
## Common SQL Query Inefficiencies

### (1) Retrieving Unnecessary Data

One of the most common inefficiencies is using:

```sql
SELECT * FROM Products;
```

This retrieves all columns, even when only a few are needed. This increases:

- Data transfer size  
- Memory usage  
- Query execution time  

#### Optimised Version

```sql
SELECT ProductName, Price FROM Products;
```

By selecting only required columns:

- Less data is processed  
- Network load is reduced  
- Query performance improves  

### (2) Unindexed JOIN Operations

JOINs combine data from multiple tables, but without indexing, the database may perform full table scans.

#### Example (Potentially Slow JOIN)

```sql
SELECT p.ProductName, o.OrderDate FROM Products p
INNER JOIN Orders o ON p.ProductId = o.ProductId;
```

If `ProductId` is not indexed in both tables:

- The database compares every row in both tables  
- Execution time increases significantly as data grows  

#### Optimisation Strategy

- Add indexes to join columns:

```sql
CREATE INDEX IX_Products_ProductId ON Products(ProductId);
CREATE INDEX IX_Orders_ProductId ON Orders(ProductId);
```

With indexing:

- The database performs index seeks instead of full scans  
- JOIN performance improves significantly  

### (3) Missing Filters (Lack of WHERE Clause)

Filtering data reduces the number of rows processed.

#### Example Without Filter

```sql
SELECT *
FROM Orders;
```

#### Optimized Version With Filter

```sql
SELECT OrderId, OrderDate
FROM Orders
WHERE OrderDate > '2023-01-01';
```

Benefits:

- Reduces dataset size  
- Improves execution speed  
- Decreases CPU and memory usage  

### (4) Inefficient Nested Subqueries

Nested subqueries may cause repeated scans of the same table.

#### Example With Subquery

```sql
SELECT *
FROM Products
WHERE ProductId IN (
    SELECT ProductId
    FROM Orders
    WHERE OrderDate > '2023-01-01'
);
```

This may:

- Trigger multiple scans  
- Increase execution time  

#### Optimised Version Using JOIN

```sql
SELECT p.*
FROM Products p
INNER JOIN Orders o ON p.ProductId = o.ProductId
WHERE o.OrderDate > '2023-01-01';
```

Why this is better:

- Processes relationships in a single step  
- Avoids redundant scans  
- Improves scalability  

---
## Key SQL Optimisation Strategies

To summarise, efficient queries typically:

- Avoid `SELECT *`
- Use indexed columns for JOINs
- Apply filters using `WHERE`
- Replace subqueries with JOINs when appropriate
- Analyse execution plans

## Copilot’s Role in SQL Optimisation

Microsoft Copilot enhances SQL performance tuning by providing AI-driven insights.

### (1) Query Analysis

Copilot can:

- Detect redundant clauses  
- Identify unnecessary subqueries  
- Highlight inefficient joins  
- Suggest removing `SELECT *`  

Example:

```sql
SELECT *
FROM Orders;
```

Copilot may recommend:

- Selecting specific columns  
- Adding filters  
- Indexing frequently filtered fields  

### (2) Indexing Recommendations

Copilot analyses patterns such as:

- Frequently filtered columns  
- Common JOIN columns  
- High-cost execution plans  

Example:

```sql
CREATE INDEX IX_Orders_OrderDate ON Orders(OrderDate);
```

Benefits:

- Faster filtering  
- Reduced execution time  
- Improved large dataset performance  

### (3) Query Refactoring

Copilot can suggest:

- Replacing subqueries with JOINs  
- Simplifying complex WHERE clauses  
- Reducing redundant logic  

Example: Subquery-based query rewritten as a JOIN for improved performance.

### (4) Execution Plan Guidance

Copilot can help interpret:

- Table scans vs. index seeks  
- Missing index warnings  
- High-cost operations  

This allows developers to:

- Identify bottlenecks  
- Measure improvements  
- Refine indexing strategies
