## Overview

-	Efficient data management is fundamental in software development  
-	Linear data structures organize data sequentially  
-	Core structures covered:
	-	Arrays  
	-	Linked Lists  
	-	Stacks  
	-	Queues  
-	Each structure has distinct strengths, memory models, and ideal use cases  

---
## Arrays and Linked Lists

-	Both store sequential data  
-	Differ mainly in:
	-	Memory allocation  
	-	Access patterns  
	-	Insertion and deletion efficiency  

### Arrays

-	Stored in a **contiguous memory layout**
	-	All elements occupy a single uninterrupted memory block  
	-	Elements are placed directly next to each other  
-	Use an **Array Memory Model**
	-	Defines how data is represented, allocated, and accessed in memory  
-	Enable **constant time access – O(1)**
	-	Direct access via index  
	-	Access time does not increase with array size  
-	Strengths:
	-	Fast random access  
	-	Simple structure  
-	Limitations:
	-	Fixed size (in many implementations)  
-	Resizing requires allocating new memory and copying elements  
-	Insertions/deletions (except at end) may require shifting elements  
-	Best suited for:
	-	Frequent lookups  
	-	Scenarios where size is known or rarely changes  

### Linked Lists

-	Composed of **nodes**
	-	Each node contains:
		-	Data  
		-	Pointer to the next node  
-	Elements are not stored contiguously  
	-	Each node can exist anywhere in memory  
-	Enable **linear time access – O(n)**
	-	Must traverse node by node to reach a position  
	-	Access time increases proportionally with size  
-	Strengths:
	-	Dynamic memory allocation  
	-	Efficient insertions and deletions  
	-	No shifting of elements required  
-	Limitations:
	-	Slower lookups compared to arrays  
	-	Extra memory required for pointers  
-	Best suited for:
	-	Frequent insertions and deletions  
	-	Scenarios with unpredictable or changing size

## Stacks and Queues

-	Abstract data types defining specific ordering principles  
-	Widely used in real-world software systems  
-	Primarily differ in how elements are added and removed  

### Stacks (LIFO – Last In, First Out)

-	Elements added and removed from the **top**  
-	Core idea:
	-	The most recently added element is processed first  
	-       Follows **Last In, First Out principle**  
-	Common use cases:
	-	Undo functionality in applications  
		-	Each action is pushed onto a stack  
		-	Undo removes the most recent action first  
-	Web browser history  
	-	Each visited page is pushed onto a stack  
	-	“Back” removes the most recent page  
-	Best suited for:
	-	Backtracking  
	-	Reversing actions  
	-	Managing nested operations  

### Queues (FIFO – First In, First Out)

-	Elements added at the **rear** and removed from the **front**  
-	Core idea:
	-	Elements are processed in arrival order  
	-	Follows **First In, First Out principle**  
-	Common use cases:
	-	Printer job management  (Print jobs processed in submission order )
-	Task queues  
	-	Background jobs (e.g., email sending, video encoding)  
	-	Prevents system overload  
-	API request management  
	-	Incoming requests handled sequentially  
	-	Ensures fairness and system stability  
-	Best suited for:
	-	Task scheduling  
	-	Request handling  
	-	Fair and ordered processing systems  

---
## Key Takeaways

-	Arrays:
	-	Fast lookups (O(1))  
	-	Less flexible resizing  
-	Linked Lists:
	-	Slower access (O(n))  
	-	Flexible and efficient modifications  
-	Stacks:
	-	Support LIFO operations  
	-	Ideal for undo and backtracking  
-	Queues:
	-	Support FIFO operations  
	-	Ideal for orderly task and request processing  
-	Choosing the correct data structure:
	-	Depends on access frequency  
	-	Depends on modification frequency  
	-	Directly impacts performance and scalability
