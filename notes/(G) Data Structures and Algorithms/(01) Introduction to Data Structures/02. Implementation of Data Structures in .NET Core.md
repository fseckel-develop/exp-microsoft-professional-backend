## Arrays and Linked Lists in .NET Core

- Both used to store collections of data  
- Differ in flexibility, memory behaviour, and modification capabilities  
- Example context: managing a deck of cards in a console application  

### Arrays

- Defined with fixed size and initialised upfront  
	- Example: `string[] cardDeck = { "Card1", "Card2", ... }` 
- Stored in contiguous memory  
- Elements accessed directly via index  
- Traversal commonly performed using a foreach loop  
- Characteristics:
	- Fast access (O(1))  
	- Simple implementation  
	- Limited flexibility after initialisation  
- Limitations:
	- Cannot easily add or remove elements after creation  
	- Insertions/deletions require shifting (O(n))  
- Best suited for:
	- Static datasets  
	- Scenarios requiring frequent indexed access  

### Linked Lists

- Implemented using `LinkedList<T>` in .NET Core  
- Elements added individually using methods like:
	- `AddLast()`
- Elements removed using:
	- `Remove()`
- Not stored contiguously in memory  
- Characteristics:
	- Dynamic size  
	- Efficient insertion and removal  
- Demonstrated behavior:
	- Add items one at a time  
	- Remove a specific item (e.g., first card)  
	- Print updated list after modification  
- Performance:
	- Access: O(n)  
	- Insertion/Deletion: O(1) if node reference known  
- Best suited for:
	- Dynamic datasets  
	- Frequent additions and removals  

### Key Comparison:

- Arrays → Faster but less flexible  
- Linked Lists → More flexible but slower access

## Stacks and Queues in .NET Core

- Both represent ordered data structures  
- Differ in how elements are added and removed  
- Demonstrated via console methods for push/pop and enqueue/dequeue  

### Stacks (LIFO – Last In, First Out)

- Implemented using `Stack<T>`  
- Key operations:
	- `Push()` → Add element to top (O(1))  
	- `Pop()` → Remove top element (O(1))  
- Behavior:
	- Last inserted element removed first  
	- Each pop reduces stack size  
- Example scenario:
	- Undo functionality  
	- Action tracking system  
- Best suited for:
	- Reversing actions  
	- Managing nested or sequential operations  

### Queues (FIFO – First In, First Out)

- Implemented using Queue< T >  
- Key operations:
	- `Enqueue()` → Add element to back (O(1))  
	- `Dequeue()` → Remove front element (O(1))  
- Behavior:
	- Elements processed in insertion order  
	- Loop continues until queue count is zero  
- Example scenario:
	- Print job management  
	- Request processing systems  
- Best suited for:
	- Fair task scheduling  
	- Order-preserving processing  

---
## Key Takeaways

- Arrays:
	- Fast access  
	- Limited modification flexibility  
- Linked Lists:
	- Flexible and dynamic  
	- Slower access due to traversal  
- Stacks:
	- Support LIFO behaviour  
	- Ideal for undo and action tracking  
- Queues:
	- Support FIFO behaviour  
	- Ideal for fair, ordered processing  
- Overall Principle:
	- Data structure selection depends on:
		- Access requirements  
		- Modification frequency  
		- Application context  
	- Choosing appropriately improves performance and scalability
