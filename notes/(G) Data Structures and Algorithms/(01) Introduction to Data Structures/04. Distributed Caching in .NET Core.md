## Overview

**Purpose of Distributed Caching**:
-	Improve application speed and scalability  
-	Reduce database load under high traffic  
-	Ensure consistent and reliable user experience  

**Core Idea**:
-	Store frequently accessed data across multiple servers  
-	Minimise latency and improve response time  

---
## Core Concepts of Distributed Caching

### Distributed Cache

-	Stores cached data across multiple servers  
-	Reduces latency by serving users from different geographic locations  
-	Improves reliability and scalability  

### Cache Hit

-	Requested data is found in the cache  
-	Result:
	-	Data retrieved instantly without querying the database  
	-	Reduced response time  
	-	Lower server load  
-	Improved scalability and user experience  

### Cache Miss

-	Requested data is not found in the cache  
-	Result:
	-	Application retrieves data from the database  
	-	Increased response time  
	-	Higher server load  
-	Follow-up: Data is stored in cache for future requests  

### Cache Invalidation

-	Process of updating or removing outdated cached data  
-	Purpose: Maintain consistency between cache and database  
-	Importance: Critical in distributed systems to ensure synchronised data  
-	Example: Product price or availability updates  

### Horizontal Scaling

-	Adding more servers to handle increasing demand  
-	Relationship to distributed caching:
	-	Each server contributes to workload distribution  
	-	Supports growing traffic without performance degradation  
-	Result:
	-	Faster response times  
	-	Improved reliability under heavy load

## Distributed Caching Tools in .NET Core

-	Common tools:
	-	Redis  
	-	SQL Server Caching  
-	Selection depends on:
	-	Data volatility  
	-	Consistency requirements  
	-	Performance needs  

### Redis

-	In-memory distributed cache (RAM-based storage)  
-	Strengths:
	-	Extremely fast data access  
	-	Ideal for frequently changing data  
-	Common Use Cases:
	-	User session management  
	-	Authentication state  
	-	Real-time stock availability  
	-	Personalized recommendations  
-	Best suited for:
	-	Temporary or dynamic data  
	-	High-speed access requirements  

### SQL Server Caching

-	Database-integrated caching mechanism  
-	Strengths:
	-	Strong data consistency  
	-	Reliable for stable data  
-	Common Use Cases:
	-	Product catalogs  
	-	Item descriptions and pricing  
	-	Reference tables  
	-	Policies and FAQs  
-	Best suited for:
	-	Stable data that changes infrequently  
	-	Applications tightly integrated with SQL databases  


---
## Combined Strategy for Scalability

-	Using Redis and SQL Server Caching together:
	-	Redis: Handles session and dynamic data  
	-	SQL Server Caching: Handles stable and structured data  
-	Benefits:
	-	Balanced performance and consistency  
	-	Reduced database strain  
	-	Faster page loads globally  
-	Outcome:
	-	Scalable, responsive, and reliable applications  


---
## Key Takeaways

-	Distributed caching improves scalability and performance  
-	Cache hits reduce latency and database load  
-	Cache misses trigger database retrieval and future caching  
-	Cache invalidation ensures data accuracy  
-	Horizontal scaling enhances system capacity  
-	Redis excels at dynamic, in-memory caching  
-	SQL Server Caching excels at stable, consistent data storage  
-	Strategic tool selection ensures optimal application performance
