## Core Algorithm Strategies

- All are comparison-based sorting algorithms  
- All reorder elements into ascending or defined sorted order  
- Differ in strategy, scalability, memory usage, and stability  

### Bubble Sort

- Iterative approach  
- Repeated adjacent comparisons  
- Gradually “bubbles” largest elements to the end  
- Conceptually simple but inefficient at scale  

### QuickSort

- Divide-and-conquer strategy  
- Selects a pivot element  
- Partitions array around pivot  
- Recursively sorts partitions  
- In-place (with recursive stack usage)  

### Merge Sort

- Divide-and-conquer strategy  
- Recursively splits dataset completely  
- Merges sorted halves  
- Requires auxiliary memory for merging  

---
## Time Complexity Comparison

- Bubble Sort  
	- O(n²)  
	- Performance degrades rapidly as dataset grows  
	- Poor scalability  

- QuickSort  
	- Average: O(n log n)  
	- Worst case: O(n²)  
	- Highly efficient in practice with good pivot selection  

- Merge Sort  
	- Guaranteed O(n log n)  
	- Predictable performance  
	- Strong scalability  

### Scaling Insight

- Small datasets → differences less noticeable  
- Medium datasets → Bubble Sort becomes inefficient  
- Large datasets → QuickSort and Merge Sort dominate  
- Quadratic growth (n²) becomes impractical quickly  
- Logarithmic divide-and-conquer scales efficiently  

---
## Space Complexity Comparison

- Bubble Sort → O(1)  
	- In-place  
	- Minimal memory usage  

- QuickSort → O(log n)  
	- Recursive call stack usage  
	- Generally memory efficient  

- Merge Sort → O(n)  
	- Requires additional memory for merging  
	- Higher memory footprint  

### Memory Trade-Off Pattern

Memory usage increases:

Bubble → QuickSort → Merge Sort  

---
## Stability Comparison

- Bubble Sort → Stable  
- QuickSort → Not stable (by default)  
- Merge Sort → Stable  

### Why Stability Matters

- Required when sorting by multiple criteria  
- Preserves relative order of equal elements  
- Important in structured datasets (e.g., records sorted by date, then ID)

## Decision Framework for Back-End Systems

### Factor 1: Task Complexity

- Simple tasks  
	- Small debug lists  
	- Lightweight operations  
	→ Bubble Sort acceptable  

- Complex data workflows  
	- User logs  
	- Large record systems  
	→ Prefer Merge Sort or QuickSort  

### Factor 2: Data Size

- Very small datasets  
	→ Any algorithm works  

- Small-to-medium datasets  
	→ QuickSort often fastest in practice  

- Large datasets  
	→ Merge Sort (consistent)  
	→ QuickSort (fast average performance)  

### Factor 3: Resource Constraints

- Memory constrained environments  
	→ Bubble Sort (O(1))  
	→ QuickSort (O(log n))  

- Memory available, large datasets  
	→ Merge Sort  

### Factor 4: Real-Time vs Batch Processing

- Real-time sorting (live updates, inventory systems)  
	→ QuickSort  
	- Fast average case  
	- Good memory balance  

- Batch processing (reports, nightly processing)  
	→ Merge Sort  
	- Stable  
	- Predictable performance  

### Factor 5: Stability Requirement

- If stability required  
	→ Merge Sort  
	→ Bubble Sort (small datasets)  

- If stability not required  
	→ QuickSort preferred for performance  

---
## Practical Back-End Use Cases

- Small business employee list  
	→ Bubble Sort  
	- Simplicity over performance  

- E-commerce product sorting  
	→ QuickSort  
	- Fast general-purpose sorting  

- Telemedicine or large record systems  
	→ Merge Sort  
	- Stable + large-scale reliability  

- Log processing systems  
	→ Merge Sort  
	- Maintains chronological consistency

2. QuickSort

	- Strong real-world default  

3. Merge Sort  
	- Stability + guaranteed performance  
	- Scalable and predictable
