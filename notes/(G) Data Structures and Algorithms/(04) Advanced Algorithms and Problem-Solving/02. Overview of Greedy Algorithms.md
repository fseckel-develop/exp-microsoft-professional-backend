## Introduction

In backend development, algorithm selection directly impacts performance and scalability. Greedy algorithms are a powerful problem-solving strategy that focuses on making **locally optimal decisions** at each step, without reconsidering previous choices or evaluating future consequences.
Unlike more comprehensive optimisation strategies, greedy algorithms do not attempt to evaluate every possible combination. Instead, they make the best immediate choice and move forward. This often leads to fast and efficient solutions — though not always globally optimal ones.

---
## What is a Greedy Algorithm?

- A greedy algorithm ...
	- makes the best possible decision at the current step  
	- does not reconsider previous decisions  
	- does not evaluate future consequences  
	- builds a solution incrementally  

- It follows a **locally optimal strategy** (each step chooses what appears best at that moment)

- Greedy algorithms are especially effective when the problem has ...
	- **greedy choice property** (a local optimum leads to a global optimum)
	- **optimal substructure** (optimal solutions can be built from optimal sub-solutions)

---
## Example 1: Greedy Task Scheduling

### Scenario

- We have multiple tasks, each defined by:
	- Name  
	- Deadline (time slot)  
	- Reward value  
- Only one task can be scheduled per time slot. The goal is to maximise total reward.

| Task | Time Slot       | Reward |
| ---- | --------------- | ------ |
| A    | 2               | 100    |
| B    | 1               | 50     |
| C    | 2               | 20     |
| D    | 1               | 70     |
### Greedy Strategy

1. Sort tasks by reward (highest to lowest)
2. Select the highest-reward task first
3. Assign it to its time slot if available
4. Continue down the sorted list
5. Skip tasks whose time slots are already filled

- After sorting by reward:
	1. Task A (100)
	2. Task D (70)
	3. Task B (50)
	4. Task C (20)

- Greedy selection:
	- Assign Task A → Slot 2
	- Assign Task D → Slot 1
	- Skip B and C (slots already filled)

### Key Observation

- The algorithm ...
	- does not evaluate combinations
	- does not reconsider lower-reward tasks
	- simply selects highest reward first
- Here this works well because maximising immediate reward also maximises total reward.

## Example 2:  Finding the Shortest Path (Dijkstra’s Algorithm)

### Scenario

- Given a graph with weighted edges:
	- Start at node A
	- Find the shortest path to node D
- One of the most famous greedy algorithms is the **Dijkstra’s Algorithm**.

### Greedy Strategy

- At each step:
	1. Choose the closest unvisited node
	2. Update distances
	3. Repeat until destination is reached
- Algorithm always selects the next closest node without reconsidering previous selections.

### Why It Works

- Dijkstra’s algorithm works because:
	- The graph has non-negative weights
	- The shortest path property holds
	- A locally shortest edge contributes to the globally shortest path

### Real-World Applications

- GPS navigation systems
- Network routing
- Traffic systems
- Infrastructure optimisation

---
## Strengths of Greedy Algorithms

- Greedy algorithms offer:
	- Fast execution
	- Simple implementation
	- Low memory overhead
	- Efficiency for large datasets
	- Often O(n log n) or better

- They are ideal when:
	- Speed is critical
	- The problem satisfies greedy choice conditions
	- Approximate solutions are acceptable

---
## Limitations of Greedy Algorithms

- Greedy algorithms do not:
	- Evaluate all possible combinations
	- Backtrack decisions
	- Guarantee global optimality in all problems

- They may fail when:
	- Future consequences matter
	- Multiple constraints interact
	- Trade-offs exist between short-term and long-term gains

---
## Greedy vs. Dynamic Programming

Greedy algorithms and dynamic programming are often compared because both solve optimisation problems.

### Greedy Approach

- Makes immediate best choice
- No backtracking
- No storing of subproblem solutions
- Fast and memory efficient
- May produce suboptimal global solution

### Dynamic Programming Approach

- Breaks problem into overlapping subproblems
- Stores intermediate results (memoization or tabulation)
- Considers all possibilities systematically
- Guarantees optimal solution (when applicable)
- Uses more memory
- Typically slower than greedy

### Example Comparison

#### Shortest Path

- **Greedy Algorithm (Dijkstra)**: 
  Finds shortest path when weights are non-negative.

- **Dynamic Programming**: 
  Could evaluate multiple path combinations when constraints are complex 
  (e.g., traffic, tolls, multiple cost factors).

#### Task Scheduling

- **Greedy Algorithm**: 
  Select highest reward first.

- **Dynamic Programming**: 
  Might consider combinations of lower-reward tasks that together produce a better overall outcome under different constraints.

---
## When to Use Greedy Algorithms

- Choose greedy when:
	- The greedy choice property holds
	- The problem has optimal substructure
	- You need fast performance
	- Memory constraints are tight
	- Approximate solutions are acceptable

- Choose dynamic programming when:
	- Subproblems overlap heavily
	- Global optimality must be guaranteed
	- Multiple constraints interact
	- Future consequences must be evaluated

---
## Backend Development Context

- Greedy algorithms are useful in:
	- Network routing
	- Resource allocation
	- Load balancing heuristics
	- Scheduling systems
	- Data compression (e.g., Huffman coding)
	- Minimum spanning tree algorithms (e.g., Kruskal, Prim)

- Dynamic programming is better suited for:
	- Caching systems
	- Rate limiting optimisation
	- Complex scheduling with dependencies
	- Text similarity algorithms
	- Financial forecasting models

---
## Conclusion

Greedy algorithms are powerful tools for making fast, efficient decisions by focusing on locally optimal choices. They excel in problems like task scheduling and shortest path calculations, where immediate optimal decisions lead to globally optimal results.

However, when problems require evaluating long-term consequences or handling overlapping subproblems, dynamic programming offers a more comprehensive and reliable approach.

Understanding when to use greedy algorithms versus dynamic programming is essential for designing high-performance backend systems that are both efficient and scalable.
