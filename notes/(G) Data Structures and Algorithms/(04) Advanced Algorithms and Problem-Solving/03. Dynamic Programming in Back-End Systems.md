## Introduction

- Dynamic programming improves back-end efficiency  
	- Stores and reuses computed data  
	- Reduces redundant calculations  
- Supports optimisation in  
	- Resource scheduling  
	- Caching strategies  
	- High-demand system environments  

---
## Core Concept in Back-End Development

- Principle  
	- Break problems into smaller subproblems  
	- Store intermediate results  
	- Reuse stored data instead of recalculating  
- Benefits  
	- Faster execution  
	- Reduced server load  
	- Improved scalability  
	- More reliable performance under traffic spikes  

---
## API Rate Limiting

- Purpose  
	- Limit number of API requests per user within a time window  
	- Ensure fair usage  
	- Prevent abuse and system overload  

- How It Works  
	- Store request count per API key or IP address  
	- Check stored counter for each incoming request  
	- Compare against allowed threshold (e.g., 100 requests per minute)  

- When Limit Is Exceeded  
	- Deny request  
	- Return HTTP 429 error  
	- Provide reset information  
	- Reset counter after time window expires  

- Dynamic Programming Role  
	- Reuses stored request history  
	- Avoids recalculating request counts  
	- Enables fast validation decisions  

- Outcome  
	- Equitable resource sharing  
	- Protection against server overload  
	- Stable performance for all users  

---
## Cache Optimisation

- Purpose  
	- Store frequently accessed data for quick retrieval  
	- Reduce expensive database operations  

- Techniques  
	- Memoization  
	- Web page caching  
	- In-memory caching (e.g., Redis)  

### Memoization

- Definition  
	- Technique that stores results of expensive function calls  
	- Returns cached result when the same input occurs again  

- How It Works  
	- Function input used as cache key  
	- Computed result stored in memory  
	- Subsequent identical calls bypass computation  

- Back-End Use Cases  
	- Repeated database queries with identical parameters  
	- Complex calculations (e.g., pricing logic, analytics aggregation)  
	- Authentication/authorization result reuse within short time windows  

- Performance Impact  
	- Reduces time complexity for repeated subproblems  
	- Converts repeated costly operations into constant-time lookups  
	- Decreases CPU and database load  

- Implementation Considerations  
	- Cache invalidation strategies  
		- Time-to-live (TTL)  
		- Event-based invalidation  
	- Memory trade-offs  
		- Faster performance vs. increased memory usage  
	- Consistency vs. freshness balance  

- Relationship to Dynamic Programming  
	- Core mechanism behind top-down dynamic programming  
	- Eliminates redundant subproblem evaluations  
	- Enables scalable, efficient back-end logic

## Scheduling Algorithms

- Purpose  
	- Optimize resource utilization  
	- Minimize wait time  
	- Maximize throughput  
	- Meet deadlines  

- Mechanism  
	- Store task queues  
	- Track resource states (CPU, memory, bandwidth)  
	- Assign priorities  

- Dynamic Adaptation  
	- Reuse historical and intermediate data  
	- Continuously update task and resource information  
	- Balance workloads dynamically  

- Example: Email Server  
	- Emails placed in queue  
	- Prioritized by urgency, size, deadline  
	- High-priority emails processed first  
	- Lower-priority emails scheduled during lower load  

- Benefits  
	- Fair resource distribution  
	- Avoidance of redundancy  
	- Prevention of server overload  
	- Consistent performance under high traffic
