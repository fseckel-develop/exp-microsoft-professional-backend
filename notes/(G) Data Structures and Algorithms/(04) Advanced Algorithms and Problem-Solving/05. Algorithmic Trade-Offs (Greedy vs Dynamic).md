## Introduction

- Choosing the right algorithmic strategy is critical for backend performance and scalability  
- Dynamic programming and greedy algorithms each offer unique strengths  
- Trade-offs impact:  
	- Response time  
	- Memory usage  
	- Accuracy of results  
	- System scalability  

---
## Dynamic Programming

- Core Idea  
	- Break complex problems into overlapping subproblems  
	- Store intermediate results (memoization or tabulation)  
	- Reuse stored results to avoid redundant computation  

- Strengths  
	- Guarantees optimal solutions when optimal substructure exists  
	- Efficient for problems with repeated subproblems  
	- Systematically explores all relevant possibilities  

- Trade-Offs  
	- Higher memory usage (tables, arrays, matrices)  
	- Additional setup time  
	- Increased implementation complexity  

- Backend Use Cases  
	- API request caching (reuse identical request results)  
	- Task scheduling with constraints (deadlines, dependencies, priorities)  
	- Maximising profit or throughput under constraints  

---
## Greedy Algorithms

- Core Idea  
	- Make the best immediate (locally optimal) choice at each step  
	- Do not reconsider previous decisions  
	- Focus on speed and simplicity  

- Strengths  
	- Fast execution  
	- Minimal memory usage  
	- Easy to implement  
	- Well-suited for real-time systems  

- Trade-Offs  
	- No guarantee of globally optimal solution  
	- Can fail in complex problems requiring foresight  

- Backend Use Cases  
	- Load balancing across servers  
	- Network routing decisions  
	- File compression (e.g., combining least-used elements first)  
	- Real-time workload distribution  

---
## Key Differences

- Optimality  
	- Dynamic Programming → Guarantees global optimum (when applicable)  
	- Greedy → May produce suboptimal results  

- Memory Usage  
	- Dynamic Programming → Higher (stores intermediate results)  
	- Greedy → Low (step-by-step decisions)  

- Execution Speed  
	- Dynamic Programming → Slower due to computation and storage  
	- Greedy → Faster due to local decision-making  

- Problem Structure  
	- Dynamic Programming → Requires overlapping subproblems + optimal substructure  
	- Greedy → Requires greedy-choice property (local choices lead to global optimum)  

---
## Choosing the Right Approach

### Problem Complexity

- Use **Dynamic Programming** when:  
	- Decisions depend on previous results  
	- Subproblems overlap  
	- Multiple constraints must be considered  

- Use **Greedy Algorithms** when:  
	- Decisions are independent  
	- Local optimal choices lead to acceptable global outcomes  
	- Real-time responsiveness is required  

---
### Performance Needs

- Choose **Greedy** when:  
	- Speed is the highest priority  
	- A “good enough” solution is acceptable  
	- Low computational overhead is required  

- Choose **Dynamic Programming** when:  
	- Accuracy and guaranteed optimality are critical  
	- The cost of a suboptimal solution is high  

---
### Memory Constraints

- Dynamic Programming  
	- Requires memory for memoization or tabulation  
	- Suitable when memory resources are sufficient  

- Greedy Algorithms  
	- Typically use constant or input-proportional memory  
	- Better when memory is limited  

---
## Conclusion

- The choice between dynamic programming and greedy algorithms depends on:  
	- Problem structure  
	- Performance priorities  
	- Memory availability  
	- Need for guaranteed optimality  

- Dynamic Programming  
	- Ensures accuracy and optimal solutions  
	- Requires more time and memory  

- Greedy Algorithms  
	- Provide speed and simplicity  
	- May sacrifice optimality for efficiency  

- Selecting the appropriate strategy directly impacts how backend systems respond, perform, and scale under real-world conditions.
