## Introduction  

- Automation is essential for efficient and reliable software delivery  
- GitHub Actions enables automated CI/CD workflows directly within repositories  
- Automates build, test, deployment, and code analysis tasks  
- Reduces manual intervention and improves team productivity  
- Integrates seamlessly with repository events  

---
## Role of GitHub Actions in CI/CD  

- Platform for Continuous Integration and Continuous Deployment  
- Executes workflows in response to repository events  
- Ensures applications remain stable and up-to-date  
- Supports automated validation before deployment  
- Enhances collaboration and development consistency  

---
## Core Components of GitHub Actions  

### Workflows  

- Defined in YAML files inside `.github/workflows/`  
- Use structured, human-readable configuration syntax  
- Automate repetitive CI/CD tasks  
- Run in response to defined events  
- Ensure consistent execution of predefined steps  

### Jobs  

- Groups of steps executed on the same virtual server  
- Defined within a workflow  
- Can run sequentially or in parallel  
- Support dependency control using keywords (e.g., `needs:`)  
- Improve task organisation and scalability  

### Steps  

- Individual commands or actions within a job  
- Executed sequentially  
- Perform tasks such as:  
	- Checking out repository code  
	- Setting up runtime environments (e.g., Node.js)  
	- Installing dependencies (`npm install`)  
	- Running tests (`npm test`)  
- Enable easier debugging and maintenance  

### Events (Triggers)  

- Activities that start workflows automatically  
- Common triggers include:  
	- `push` events  
	- `pull_request` events  
	- Scheduled runs  
	- Manual triggers (`workflow_dispatch`)  
- Ensure workflows run at the correct time without manual action

## Example: Basic CI Pipeline Structure  

### Repository Preparation  

- Initialise local repository  
- Add and commit project files  
- Push to remote repository  
- Create `.github/workflows/ci.yaml` file  

### YAML Workflow Configuration  

- Define workflow name  
- Specify trigger events (push, pull requests)  
- Define jobs section  
- Choose runtime environment (e.g., Ubuntu latest)  
- Add steps to:  
	- Check out repository  
	- Set up runtime (e.g., Node.js version)  
	- Install dependencies  
	- Run automated tests  
- Automatically execute workflow on every push  

### Execution and Monitoring  

- Workflow runs automatically after code push  
- Status displayed in repository’s Actions tab  
- Successful runs indicated by completion status  
- Failed runs provide detailed logs for debugging  
- Each new push re-triggers the defined workflow

## Optimisation and Troubleshooting  

### Workflow Not Triggering  

- Verify correct event configuration  
- Confirm proper file location and naming  
- Ensure branch matches trigger condition  

### Job Failures  

- Review logs for failing steps  
- Validate environment configuration  
- Check repository secrets and dependencies  

### Performance Improvements  

- Cache dependencies for faster builds  
- Remove unnecessary steps  
- Combine or parallelise jobs strategically
