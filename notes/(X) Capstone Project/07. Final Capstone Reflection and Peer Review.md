## Project Description

### System Purpose

- `LogiTrack` is an API-driven Order and Inventory Management System designed to:
	- Manage warehouse inventory  
	- Handle customer orders and order items  
	- Enforce strict role-based access control  
	- Persist data securely using EF Core + Identity  

### Core Functional Areas

- Inventory Management
	- Full CRUD operations
	- Stock manipulation via domain methods:
		- IncreaseStock`
		- `DecreaseStock`
		- `UpdateLocation`
	- Role-protected write operations
	- Cached read endpoints

- Order Management
	- Create orders with multiple items
	- Add / remove / adjust items inside orders
	- Automatic subtotal & total calculation
	- Proper relational modelling (Order → OrderItems → InventoryItem)

- Authentication & Authorisation
	- ASP.NET Identity integration
	- JWT-based authentication
	- Email confirmation requirement
	- Role-based authorisation policies

## Architecture and Design Decisions

### Application Startup Configuration

- The system is configured in `Program.cs` with:
	- `AddDbContext<LogiTrackContext>` (SQLite)
	- `AddIdentity<ApplicationUser, IdentityRole>`
	- JWT Bearer authentication
	- Custom authorisation policies:
		- **InventoryWrite** → Admin, WarehouseStaff
		- **OrderWrite** → Admin, SalesStaff
		- **AdminOnly** → Admin
	- Automatic role seeding at startup
	- Swagger UI in development environment

- Identity password & lockout policies:
	- Minimum 8 characters
	- Uppercase, lowercase, digit, special character required
	- 5 failed attempts → 10 minute lockout
	- Unique email required
	- Confirmed email required for sign-in

### Database & Persistence Layer

- `LogiTrackContext` inherits from `IdentityDbContext<ApplicationUser>`, allowing:
	- Identity tables
	- Orders
	- InventoryItems
	- OrderItems

- Relationships
	- Many-to-Many Relationship
		- One Order can include Many InventoryItems
		- One InventoryItem can be included in Many Orders
		- OrderItem does work as Junction, to normalise the database
	- Configured explicitly using Fluent API in `OnModelCreating`

- Migrations managed via:
	- `IDesignTimeDbContextFactory`
	- CLI migration commands
	- SQLite database persistence

## Domain Modeling & Business Logic

### Entities

- **ApplicationUser**
	- Extends IdentityUser
	- Supports RefreshToken storage

- **InventoryItem**
	- Strong validation with DataAnnotations
	- Encapsulated stock manipulation logic
	- Prevents invalid stock updates

- **Order**
	- Encapsulated AddItem / RemoveItem logic
	- Merges duplicate items automatically
	- Computes `TotalAmount` dynamically

- **OrderItem**
	- Calculates `SubTotal`
	- Guards against invalid quantity updates

### Separation of Concerns

- Controllers → HTTP layer  
- DTOs → Request/Response contracts  
- Entities → Business logic  
- DbContext → Persistence  
- Identity → Authentication & user management  
- IMemoryCache → Performance layer  

- DTOs are used consistently for:
	- Create
	- Update
	- Response
	- Register
	- Login

This prevents overexposing domain models.

## API Design & Routing

### InventoryController

- Supported Routes:
	- `GET /api/inventory`
	- `GET /api/inventory/{id}`
	- `POST /api/inventory` → Policy: InventoryWrite
	- `PUT /api/inventory/{id}` → Policy: InventoryWrite
	- `DELETE /api/inventory/{id}` → Policy: AdminOnly

- Caching:
	- Cached list endpoint
	- Cached individual items
	- Automatic invalidation on write operations
	- Sliding expiration: 5 minutes

### OrdersController

- Supported Routes:
	- `GET /api/orders`
	- `GET /api/orders/{id}`
	- `POST /api/orders` → Policy: OrderWrite
	- `POST /api/orders/{orderId}/items`
	- `DELETE /api/orders/{orderId}/items/{inventoryItemId}`
	- `PATCH /api/orders/{id}`
	- `PATCH /api/orders/{id}/items/{inventoryItemId}`
	- `DELETE /api/orders/{id}` → Policy: AdminOnly

- Optimisations:
	- `.Include()` + `.ThenInclude()` for eager loading
	- `.AsNoTracking()` for read-only queries
	- Projection using expression selectors
	- Cache invalidation per order

## Security Implementation

### Authentication

- JWT Bearer tokens
- Token validation:
	- ValidateIssuer
	- ValidateAudience
	- ValidateLifetime
	- ValidateSigningKey
- Tokens passed via: `Authorization: Bearer {token}`

### Authorization

Global `[Authorize]` on controllers. Policy-based access control:

| Policy         | Roles                 |
| -------------- | --------------------- |
| InventoryWrite | Admin, WarehouseStaff |
| OrderWrite     | Admin, SalesStaff     |
| AdminOnly      | Admin                 |

This enforces strict separation of operational responsibilities.

### Identity Security Features

- Strong password enforcement
- Email confirmation required
- Account lockout after failed attempts
- Unique email constraint
- Role seeding at startup

- Outcome:
	- Unauthorised users cannot access API
	- Write operations are tightly restricted
	- Admin-only destructive operations

## Caching & Performance Optimisation

### IMemoryCache Strategy

Applied to:
- Inventory GET endpoints
- Orders GET endpoints

Features:
- Sliding expiration (5 minutes)
- Individual and collection caching
- Explicit invalidation on mutations
- Reduced repeated DB calls

### Query Optimisation

- `.AsNoTracking()` for reads
- Expression-based projection
- Eager loading only when required
- Dictionary lookup for batch inventory validation
- Reduced N+1 query risks

## Development Challenges & Solutions

### Challenge 1: Identity + JWT Integration

Solution:

Solution:

- Correct navigation properties

Solution:
- Centralised InvalidateCache methods
- Removed both collection & individual cache entries
- Used sliding expiration for balance

Solution:
- Encapsulated domain logic inside entities
- Threw exceptions for invalid operations
- Used DataAnnotations validation
