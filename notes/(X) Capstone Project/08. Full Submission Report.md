## Description of the Project

My API project, `LogiTrack`, is a secure and role-based Order and Inventory Management System built using ASP.NET Core Web API, Entity Framework Core with SQLite, and ASP.NET Core Identity. The system is designed to manage warehouse inventory, process customer orders, and enforce strict access control through authentication and authorisation mechanisms.

The API provides full CRUD functionality for inventory items. Authenticated users can retrieve inventory data, while only users with the appropriate roles—such as **Admin** or **WarehouseStaff**—are allowed to create or update inventory records. Deleting inventory items is restricted to the **Admin** role. The system also includes business logic within the InventoryItem entity to safely increase or decrease stock levels and prevent invalid operations, such as reducing stock below zero.

The order management component allows authorised users—specifically **Admin** and **SalesStaff**—to create customer orders, add or remove items, and adjust item quantities dynamically. Each order can contain multiple order items, and the total amount is calculated automatically based on item quantities and unit prices. The domain model ensures that duplicate items are merged within an order, and subtotals and totals are computed dynamically to maintain consistency.

Security is a core feature of the project. The API uses ASP.NET Identity for user management and JWT Bearer tokens for authentication. Password policies enforce strong security requirements, including minimum length, uppercase and lowercase characters, digits, and special characters. The system also requires email confirmation before login and implements account lockout after multiple failed attempts. Authorisation is implemented using policy-based role restrictions, ensuring that only users with the correct roles can perform specific operations.

Performance optimisation is achieved through the use of in-memory caching for frequently accessed GET endpoints. The API uses sliding expiration and explicit cache invalidation when data changes. Additionally, query performance is improved using techniques such as AsNoTracking() for read-only queries, eager loading with Include() and ThenInclude(), and projection into DTOs to reduce unnecessary data transfer.

The project also includes comprehensive unit testing using xUnit and FluentAssertions. These tests validate business logic, entity behaviour, subtotal and total calculations, and database interactions using an in-memory SQLite database. This ensures the reliability and correctness of the system.

Overall, `LogiTrack` demonstrates secure backend development, clean architectural separation, domain-driven design principles, and performance optimisation techniques suitable for real-world enterprise applications.

## Major Challenges

One of the main challenges I faced was integrating ASP.NET Identity with JWT authentication. Configuring the authentication middleware, token validation parameters, issuer, audience, and signing key required careful setup to ensure secure access to protected endpoints. I overcame this by properly configuring the authentication pipeline and thoroughly testing endpoints with valid and invalid tokens to confirm correct behaviour.

Another challenge was implementing role-based authorisation using policies. I needed different roles to have different permissions, such as restricting inventory write operations to Admin and WarehouseStaff and limiting order modifications to Admin and SalesStaff. I solved this by defining custom authorisation policies and applying them to specific controller actions, as well as seeding roles at application startup.

Modeling the relationship between Orders and InventoryItems was also challenging. The system required a many-to-many relationship, which I implemented by introducing OrderItem as a junction entity. This allowed each order to contain multiple inventory items while also storing additional data such as quantity and unit price. Configuring this relationship correctly using Fluent API ensured proper navigation properties and efficient queries.

Finally, maintaining cache consistency was difficult after implementing in-memory caching. To prevent stale data, I created centralised cache invalidation methods that clear both collection and individual cache entries whenever data is modified. This allowed me to improve performance without sacrificing data accuracy.

## Implementation Details

I implemented business logic directly inside the domain entities to keep controllers clean and maintain proper separation of concerns. The Order entity contains methods such as `AddItem()`and `RemoveItem()`, which handle merging duplicate items and maintaining order integrity. The InventoryItem entity includes methods like `IncreaseStock()` and `DecreaseStock()` to prevent invalid stock operations. The OrderItem entity calculates its SubTotal dynamically and validates quantity updates. This approach ensures that business rules are enforced at the model level rather than in the controllers.

For data persistence, I used Entity Framework Core with SQLite. The `LogiTrackContext` inherits from `IdentityDbContext<ApplicationUser>`, allowing both application data and Identity tables to be stored in the same database. I defined DbSet properties for Orders, InventoryItems, and OrderItems, and configured relationships explicitly using the Fluent API in `OnModelCreating()`. Database schema changes were managed through migrations, ensuring consistent and version-controlled persistence.

State management is primarily handled through the relational database, which preserves application data across restarts. Authentication state is managed using JWT tokens, which are validated on each request to maintain stateless security. Additionally, I implemented in-memory caching for frequently accessed GET endpoints. The cache uses sliding expiration and explicit invalidation after write operations to balance performance with data consistency.

## Security Measures

I implemented authentication using ASP.NET Core Identity combined with JWT Bearer tokens. Users must log in to receive a signed JWT, which is validated on every request using issuer, audience, lifetime, and signing key validation. All controllers are protected with the `[Authorize]` attribute to ensure that only authenticated users can access the API.

I enforced strong password policies by requiring a minimum length of eight characters, as well as uppercase letters, lowercase letters, digits, and non-alphanumeric characters. I also enabled account lockout after five failed login attempts for ten minutes to protect against brute-force attacks. Additionally, I required unique email addresses and mandatory email confirmation before users are allowed to sign in.

For authorisation, I implemented a policy-based RBAC pattern. I defined custom policies such as InventoryWrite, OrderWrite, and AdminOnly, which restrict certain operations to specific roles like Admin, WarehouseStaff, and SalesStaff. Destructive operations, such as deleting orders or inventory items, are limited to the Admin role only.

Finally, I used Data Annotations and model validation to prevent invalid or malicious input from being processed. This ensures that only properly validated data is persisted to the database, strengthening the overall security of the API.

## Cache Management and Performance Optimisation

I managed caching by using `IMemoryCache` for frequently accessed GET endpoints in both the Inventory and Orders controllers. I cached both collection results (such as all inventory items or all orders) and individual records. Each cache entry uses sliding expiration set to five minutes, which keeps frequently accessed data in memory while allowing unused entries to expire automatically.

To maintain consistency, I implemented centralised cache invalidation methods. Whenever a create, update, or delete operation occurs, the related collection cache and any affected individual item cache entries are explicitly removed. This ensures that stale data is not returned while still improving read performance.

I optimised database performance by using `AsNoTracking()` for read-only queries to reduce Entity Framework tracking overhead. I also used projection with expression selectors to return only the required DTO fields instead of full entities. For relational data, I applied `Include()` and `ThenInclude()` to eagerly load related entities and prevent N+1 query issues. 
Additionally, I minimised database round-trips by batching lookups, such as retrieving required inventory items in a dictionary before creating an order.
